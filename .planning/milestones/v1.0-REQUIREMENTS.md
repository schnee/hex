# Requirements: Hex Layout Toolkit

**Defined:** 2026-03-31
**Core Value:** A user can quickly generate a high-quality hex layout and confidently preview how it will look on their real wall before building.

## v1 Requirements

Requirements for the Streamlit-to-React migration release. Each requirement must map to exactly one roadmap phase.

### Migration & Platform

- [x] **MIGR-01**: User can complete generation and overlay workflows entirely in the React application without needing Streamlit screens
- [x] **MIGR-02**: User can move between generator and overlay views in a single routed React experience while keeping selected pattern context
- [x] **MIGR-03**: User receives behavior in the React app that matches backend API contracts for generation, upload, and overlay calculation endpoints

### Pattern Generation

- [x] **GEN-01**: User can set generator parameters (aspect ratio, tile counts, tendril settings, color strategy) with immediate validation feedback
- [x] **GEN-02**: User can generate multiple pattern variants in one action and view each variant as a selectable result
- [x] **GEN-03**: User can choose one generated pattern as the active overlay candidate
- [x] **GEN-04**: User can download the selected pattern as a PNG export

### Wall Overlay

- [x] **OVR-01**: User can upload a wall image via drag-and-drop or click-to-upload and gets clear rejection messages for invalid files
- [x] **OVR-02**: User can drag and resize the active pattern overlay using visible selection handles
- [x] **OVR-03**: User sees live overlay dimensions while moving or resizing, using backend-authoritative calculations
- [x] **OVR-04**: User can zoom and pan the wall canvas to place overlays precisely
- [x] **OVR-05**: User can clearly see which overlay object is selected and can deselect/reselect predictably

### Reliability & UX Feedback

- [x] **UX-01**: User sees explicit loading, success, and actionable error states for generation, upload, and overlay operations
- [x] **UX-02**: User experiences stable interactions (no obvious flicker, jump, or stale state) during core generator and overlay tasks

## v2 Requirements

Deferred differentiators and enhancements after parity release.

### Enhanced Editing

- **ENH-01**: User can undo and redo overlay edits
- **ENH-02**: User gets snapping and alignment guides during overlay placement
- **ENH-03**: User can auto-place pattern with a one-click best-fit action
- **ENH-04**: User can compare before/after placement states in a dedicated comparison mode

### Persistence & Productivity

- **PERS-01**: User can save and reload design sessions (generator settings + overlay placement)
- **PERS-02**: User can start from preset layout recipes for common room scenarios

## Out of Scope

Explicit exclusions for this migration milestone.

| Feature | Reason |
|---------|--------|
| Full 3D room modeling/photoreal rendering | High complexity and not required to deliver parity-first migration |
| Multi-user accounts and collaboration | Adds auth/state complexity beyond current single-user workflow objective |
| CAD/BIM import workflows | Large integration surface not needed for core wall-photo overlay use case |
| AI full-room auto-design generation | Distracts from deterministic layout/overlay quality and migration scope |
| Native mobile applications | Web migration is current priority |
